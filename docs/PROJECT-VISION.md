# Project Vision

## One-paragraph pitch

**Restaurant Platform** — это full-stack pet-проект, представляющий собой платформу для управления небольшой сетью ресторанов. Состоит из двух поверхностей: (1) публичная витрина для гостей с просмотром меню, бронированием столов и оформлением заказов навынос — рендерится на сервере для SEO; (2) внутренний админ-дашборд для управляющего сети с real-time-доской заказов, аналитикой по выручке/блюдам и редактированием меню. Цель проекта — продемонстрировать владение современным Angular (v20+, signals, zoneless, SSR/Incremental Hydration, новый control flow, `@defer`) и параллельно — full-stack-стеком (NestJS + MongoDB в Nx-монорепо).

## Target audience

- **Основная**: рекрутёры и tech-leads, оценивающие резюме на позиции Angular / full-stack Angular+Node.
- **Вторичная**: я сама — упражнение по сборке полноценной системы end-to-end, от схемы БД до деплоя.

## Goals

1. **Резюме**: жирный bullet «Angular 20 + NestJS + MongoDB monorepo, SSR with Incremental Hydration, signals, SSE, deployed».
2. **Закрепление знаний**: применить материал из `angular-deep-dive.md` и `nestjs-deep-dive.md` на одном связном продукте.
3. **Портфолио-демо**: живой URL, который можно прислать вместе с резюме.

## In scope (что делаем)

- Публичный сайт ресторана (меню, информация, бронь, заказ навынос) с SSR.
- Аутентификация: гость → регистрация → залогиненный пользователь; отдельная роль admin.
- Корзина и оформление заказа с шагом «оплата» (заглушка, без реальных платежей).
- Real-time через **SSE**: статус заказа у пользователя, доска новых заказов у админа.
- Админ-дашборд: CRUD меню/блюд, список заказов, простая аналитика (выручка по дням, топ-блюд).
- i18n (ru/en).
- Тёмная и светлая темы.
- Адаптивная вёрстка (mobile / tablet / desktop).
- Деплой на Render (фронт + бэк) + MongoDB Atlas M0.
- На финальной итерации: добавить **WebSocket** для двунаправленного сценария (например, «официант пингует гостя — стол готов / заказ принят»).

## Out of scope (что НЕ делаем)

- Реальные платежи (Stripe и прочие) — только UI-заглушка.
- Реальная доставка с курьерами и геолокацией — только «навынос».
- Множество ресторанов с правами на уровне ресурсов — у админа доступ ко всему.
- Push-уведомления, email, SMS — никаких внешних сервисов рассылки.
- Мобильное приложение (только web, адаптивное).
- Сложная аналитика (cohorts, funnels) — только базовые графики.
- Multi-tenancy, white-label.
- OAuth (Google/Apple) — только email+password.
- Тесты «на 100% coverage» — точечно: критичные кейсы на unit, 2-3 e2e на главный flow.

## Tech stack

| Слой | Технология | Зачем |
|---|---|---|
| Монорепо | **Nx** | Стандарт для Angular+Nest, shared libs, generators |
| Frontend | **Angular 20+** standalone, signals, zoneless, SSR | Главная цель проекта |
| UI Kit | **Angular Material** + кастомные SCSS-токены | Знакомо пользователю, быстрый старт |
| State | **Signals** + **`@ngrx/signals` Signal Store** | Голые сигналы для локальной логики; Signal Store для глобальных stores (Cart, Auth, Menu, Orders). Даёт DevTools, конвенции, плагины (`withEntities`, persistence) без Redux-boilerplate'а классического NgRx |
| Reactive primitives | `linkedSignal`, `computed`, `effect`, `resource`/`httpResource` | Производное состояние, side-effects, HTTP-как-signal |
| Real-time | **SSE** (нативный `EventSource`) | Подходит для one-way сервер→клиент |
| Backend | **NestJS** | Знакомый пользователю стек, естественен с Angular |
| Database | **MongoDB + Mongoose** | Пользователь уже знает Mongoose |
| Auth | **JWT** access + refresh, httpOnly cookie | Минимум, который покрывает безопасность |
| Тесты | Jest (unit) + Playwright (e2e, 2-3 smoke) | Минимум для демонстрации |
| Деплой | **Render** (web services) + **MongoDB Atlas M0** | Бесплатно, поддерживает SSR и persistent connections |
| CI | GitHub Actions: lint + test + build | Не более 2-3 простых job'ов |

## Success criteria

Проект считается «готовым к показу», если:

1. Гость может зайти на публичный URL, увидеть меню (отдано из SSR), положить блюдо в корзину, зарегистрироваться, оплатить (заглушкой) и увидеть статус заказа в реальном времени.
2. Админ может залогиниться, увидеть новый заказ моментально (без обновления), отредактировать блюдо, увидеть базовый график выручки.
3. Lighthouse Performance ≥ 85, Accessibility ≥ 90 на главной странице.
4. README с архитектурной диаграммой, скриншотами, ссылкой на живой деплой.

## Constraints

- **~10 часов в неделю**, без жёсткого дедлайна.
- Сначала **простой скелет** — потом фичи итеративно.
- WS добавляется только в финальной итерации.
- Никакого преждевременного усложнения: одна продакшен-фича вытягивается до конца, прежде чем браться за следующую.
